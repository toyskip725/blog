ブログを自作して、はてなブログから移行した。

# 技術スタック

どう作るか検討する上で、いくつか考慮事項があった。
- ほぼ間違いなく、将来のどこかの時点で再びブログのリプレイスをしたくなるはずなのだけれど、そのときにもブログ記事データはそのまま移行できるようにしたい。
  - リプレイスする主要因はおそらくデザイン上の問題なので、UIデザイン部分の実装は今後数年程度メンテナンスできるものであればよい。
  - ただし、不必要に移行難易度が上がるのは嫌なので、依存するライブラリはできるだけ少なくする。
- 想定しているブログの基本的用途は純粋に個人的なメモを残すことであり、コメントを管理したり、アクセス数を伸ばしたりする仕組みはいらない。どちらかというと、Visual Studio Code + Gitという使い慣れた道具で、スムーズに記事を書ける体験を重視したい（はてなブログのエディタは結局今に至るまで使いこなせていない）。
- 開発やメンテナンスに時間がかからない方がよいので、JavaScript/TypeScriptベースで開発したい。

静的サイトジェネレータは[Astro](https://astro.build/)を使用した。
[notebooks](https://toyskip725.github.io/notebooks/)もAstroで動いており少し慣れているのと、同時にメンテナンスできて効率的かもと思ったので。

ブログ記事はマークダウンで作成していく方式にした。
Astro組み込みのマークダウン処理系でやりたいことはだいたいできるので、MDXは導入していない。
あとは単に依存先を増やしたくないのもある。

UIコンポーネント部分はAstroの`@astrojs/lit`インテグレーションを使用した。
最近はReactかSvelteのどちらかを使用することが多かったので、それ以外でかつ今まで触ったことのないライブラリを使おうと思った。
PreactかSolidJSを使おうともしていたのだけれど、今までの経験からほぼJSXを書くだけで終わりそうなので、Reactと開発体験が変わらずつまらなくなりそうと考え直して今回は不採用とした。


# 作業中の気づき

Astroには、ファイルベースルーティングとコンテンツコレクションという二通りのマークダウンの取り扱いがある。
ファイルベースルーティングでも大きな問題はないと思っていたのだけれど、このやり方にはそれぞれの`.md`ファイルがレイアウトとルーティングパスに密結合してしまう、という問題があることに気づいた。
今回コンテンツコレクションで管理することでそのあたりの見通しは少しよくなった。
TypeScriptの型まわりでも恩恵があるはずなのだけれど、まだあまりよくわかっていない。

コードブロックについてはAstro組み込みのシンタックスハイライト（内部的にはShiki）があるけれど、
インラインのコード（例えば `int` `boolean`のようなやつ）についてはスタイルを自分で書く必要があった。
最終的にそこそこの見映えにできたと思うので、これで技術記事などを書いてもよいかもしれない。

``` typescript
// コードブロックの例
async function fetchAllTitle() {
  const posts = await fetchAll();
  return posts.map((post) => post.title);
}
```
